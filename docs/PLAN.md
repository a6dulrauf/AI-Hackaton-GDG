# LifeLine — Build Plan
## Al-Khidmat Problem 1: Real-Time Blood Donor Matching

**Stack:** FastAPI (Python) backend + React (Vite) frontend
**AI:** Groq (chat parsing + free Whisper voice) · Gemini (reserved for Problem-2 OCR fallback)
**Channel:** WhatsApp-style chat UI (mocked — no real WhatsApp sending)
**Timebox:** 5–6 hours on the clock (+ a short pre-hackathon prep phase)
**Posture:** Ship the core flow first. Stretch features are optional. Problem 2 is the scope-expansion backup.

> Companion files: `CLAUDE.md` (harness/context), `WINNING_CONCEPT.md` (positioning + pitch),
> `data/generate_donors.py` (dataset), `backend/ranking.py` (matching brain).

---

## 1. Scope — committed vs not

**Committed core (must work end-to-end):** a requester types a plain-language blood request;
the system extracts fields, ranks eligible nearby donors, explains *why* each one, "contacts"
them in waves, protects the donor network from over-asking, and shows a coordinator a live
confirmation count + final summary.

**Optional stretch (only if core is green):** Urdu voice input, blood-group compatibility
fallback toggle, map dispatch animation, donor-side free-text reply interpretation.

**Out of scope (say "roadmap", never build):** real WhatsApp/SMS, database, auth, scale.

## 2. Core features (the committed MVP)

| # | Feature | What it does |
|---|---------|--------------|
| C1 | Natural-language intake | LLM turns a mixed Urdu/Roman-Urdu/English message into structured fields (group, count, location, hospital, urgency) + asks only for what's missing |
| C2 | Eligibility + ranking | Gate ineligible donors; score the rest by proximity + recency + response history − fatigue |
| C3 | Chat + donor list | WhatsApp-style chat; ranked donor list |
| C4 | Map | Leaflet + OSM: hospital marker + donor pins colored by status |
| C5 | Wave outreach + live status | Contact a wave of top donors; track commitments; auto-escalate; "X of Y confirmed"; final summary |
| C6 | "Why this donor" rationale | Plain-English reason per donor (from `ranking.py` `why` field), not just a score |
| C7 | Network-health + Spam Shield | Visibly skip/label recently-contacted donors; show "a blast would message N, we contacted M" |

## 6. Data model

**donors.csv columns:** `donor_id, name, gender, dob, age, blood_group, phone, neighbourhood,
lat, lng, last_donation_date, days_since_last_donation, total_donations,
times_contacted_last_30d, response_rate` (generated by `data/generate_donors.py`).

**Hospitals:** provided in `ranking.py` (`HOSPITALS` + `resolve_hospital()` with alias/substring match).

**Runtime objects:**
```
Request   = {id, raw_text, blood_group, count_needed, location_text, hospital,
             hospital_latlng, urgency, missing_fields[]}
DonorState= {donor_id, score, distance_km, eligible, reason, why, wave, status}
            status ∈ {pending, contacted, confirmed, declined, ineligible}
```

---

## 7. The pieces that need care

### 7.1 Ranking (in `ranking.py` — reuse, don't reinvent)
- Eligibility: age 18–60; last donation > 90d (men) / > 120d (women).
- Default exact blood-group match; `allow_compatible=True` uses recipient-compatibility map.
- `score = 0.40*proximity + 0.25*recency + 0.20*response_rate − 0.15*fatigue`.
- Emits both `reason` (chips) and `why` (sentence for C6).

### 7.2 LLM intake (Groq — `llm.py`)
- Strict JSON: `{blood_group, count, location, hospital, urgency, missing_fields}`.
- 3–4 few-shot examples mixing Urdu / Roman Urdu / English.
- Regex fallback so the endpoint never 500s on a rate-limit.
- Model: `llama-3.3-70b-versatile`.

### 7.3 Wave engine
- Wave size 8–10; top-N eligible not yet contacted.
- Demo control: a manual **"Advance wave"** button + DonorSim buttons (more reliable on stage than timers).
- When `confirmed >= count_needed`, render the summary.
- C7: track how many *matching* donors exist vs. how many were contacted → Spam Shield.

## 8. Backend endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/donors` | list donors (for the map) |
| POST | `/request/parse` | raw text → structured fields + missing_fields |
| POST | `/request/create` | create request, rank, mark wave-1 contacted; return donors + Spam-Shield counts |
| GET | `/request/{id}/status` | wave, confirmed count, donor states (poll every 2s) |
| POST | `/request/{id}/escalate` | advance to next wave |
| POST | `/donor/respond` | {donor_id, text} → classify confirm/decline/eligibility-update |
| POST | `/voice` *(optional)* | audio → Whisper → reuse /request/parse |

## 11. Demo script (≈3 min)

1. Type **"AB negative chahiye jaldi, Liaquat National mein"** → fields pop as chips.
2. Answer the one missing field (count) → "3".
3. Map lights up; read one **why** sentence aloud.
4. Wave 1 dispatches → counter ticks → simulate 2 confirms + 1 "gave blood last month" (eligibility update, not refusal).
5. Auto-escalate to wave 2 → reach "3 of 3 confirmed".
6. Point at **Spam Shield**: "a blast would have messaged 240; we contacted 9." Show summary.

## 13. Risks & mitigations

| Risk | Mitigation |
|------|------------|
| LLM rate limit/latency mid-demo | cache last good parse; regex fallback; warm cache once before judging |
| Live timing feels random | manual "Advance wave" + DonorSim buttons instead of timers |
| Map tiles slow/blocked | pins still render without tiles |
| Scope creep | lock C1–C7 first; stretch only when green |
| Hospital name not found | `resolve_hospital` aliases + "pick from list" fallback |
